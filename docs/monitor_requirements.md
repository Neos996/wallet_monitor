# 需求与范围说明

本文档用于说明 `wallet_monitor` 要解决什么问题、当前已经做到什么程度、明确不做什么，以及后续应如何演进。

## 1. 目标定义

`wallet_monitor` 的目标不是把扫链逻辑嵌入业务系统，而是提供一个**独立部署的钱包入账监控服务**。

业务系统只负责两件事：
1. 注册需要监控的收款地址。
2. 接收入账回调并完成自身业务处理。

监控服务负责以下职责：
- 周期性扫链
- 按确认数判断入账是否成立
- 对同一笔入账做幂等去重
- 将回调任务持久化
- 对失败回调自动重试并保留死信

## 2. 当前明确支持的范围

### 2.1 链与资产范围
当前版本明确支持：
- `chain=mock`：本地闭环联调
- `chain=tron`：`mainnet | shasta | nile` 的已确认 `TRX(native)` / `TRC20`
- `chain=evm`：`ERC20` 入账扫描

### 2.2 核心能力范围
当前版本已经具备：
- 地址管理接口
- 单地址或默认回调地址配置
- 定时扫描与手动触发扫描
- 最小确认数控制
- 持久化回调任务队列
- 指数退避重试
- 死信任务查询、导出、重试
- 成功回调后的最终去重
- 管理接口鉴权
- 回调签名
- Prometheus 指标

### 2.3 当前不承诺的范围
当前版本**不承诺**以下能力：
- 多实例高可用
- 共享数据库下的分布式调度
- 深度链重组恢复
- BTC 等其他链适配
- 复杂智能合约事件语义识别
- SaaS 多租户隔离

因此，当前系统的准确定位是：

**单机部署、可落地使用的钱包入账监控服务。**

不是纯 Demo，也不是最终态的多链高可用索引平台。

## 3. 当前实现结论

### 3.1 已经完成的部分
当前代码已经完成以下核心闭环：

#### 地址管理
- `GET /addresses`
- `POST /addresses`
- `GET /addresses/{id}`
- `PATCH /addresses/{id}`
- `DELETE /addresses/{id}`
- `POST /addresses/{id}/enable`
- `POST /addresses/{id}/disable`

#### 扫描与进度推进
- 周期性扫描启用中的监控地址
- 支持 `POST /scan/once` 手动触发
- 每个地址维护 `last_seen_height`
- 只处理达到确认数的交易

#### 回调与幂等
- 将回调写入 `CallbackTask`
- 自动重试失败回调
- 超过重试次数进入 `dead`
- 回调成功后写入 `ProcessedTx`
- EVM 按 `tx_hash + log_index` 做事件唯一标识

#### 调试与验证
- `mock` 闭环联调
- TRON 真链联调
- EVM 本地 ERC20 验证

### 3.2 推荐对外表述
对内或向管理层汇报时，建议统一使用以下表述：

**当前系统已经具备单机部署条件，能够完成地址注册、扫链、确认数判断、幂等去重、回调重试和死信处理。现阶段重点不是“从 0 到 1 是否可用”，而是“如何从单机 MVP 继续演进到更高规模和更高可用形态”。**

## 4. 当前架构判断

### 4.1 当前主方案
当前系统主方案是：

- 以“监控地址”为核心对象管理扫描进度；
- 根据链类型调用不同适配器扫描新入账；
- 扫描命中后生成统一 `Tx` 与统一回调 payload；
- 通过数据库保证任务持久化和幂等。

### 4.2 EVM 的两种扫描模式
EVM 当前支持两种模式：
- `address`：按地址逐个扫日志
- `block`：按区块高度推进、批量合并 watched 地址查询

因此当前系统不是“只能按地址轮询”的结构。更准确地说：

- TRON 当前主要是按地址查询交易历史；
- EVM 已经支持按区块日志聚合扫描；
- 整体架构允许后续继续往更强的索引能力演进。

## 5. 风险边界

### 5.1 当前主要工程风险
- SQLite 只适合单机，不适合多实例并发写入
- 依赖外部 RPC / TronGrid，可受限于限流和稳定性
- 当前未覆盖极端深 reorg 处理
- 管理接口与调试接口需要严格依赖网络隔离和 token 保护

### 5.2 当前安全要求
生产部署至少应满足：
- 开启 `-admin-token`
- 开启 `-callback-secret`
- 管理接口仅对内网开放
- 业务侧必须做回调验签和幂等
- 监控 `/metrics` 与死信任务

详见 [SECURITY.md](./SECURITY.md)。

## 6. 后续演进优先级

### P0：现网可用性增强
- 强化部署基线与上线检查清单
- 强化监控、告警和排障手册
- 强化 callback_url 管控和安全限制
- 强化 RPC 限流、超时、失败分类

### P1：规模化能力增强
- 从 SQLite 演进到共享数据库
- 支持多实例部署与分布式协调
- 增加批量地址管理能力
- 增加操作审计能力

### P2：多链与更强索引能力
- 扩展更多链适配
- 对更多资产类型做标准化处理
- 增强 reorg 回滚与补偿机制
- 逐步演进为更通用的链上索引服务

## 7. 一句话结论

`wallet_monitor` 当前已经完成“地址注册 → 扫链 → 去重 → 回调 → 重试 → 死信”的完整闭环，适合作为单机版入账监控服务落地使用；后续工作的重点是提升规模、可用性和多链覆盖，而不是重做基础链路。
