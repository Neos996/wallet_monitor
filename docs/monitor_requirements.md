# 监控服务需求整理

## 1. 背景

该监控服务用于支付收款场景，目标是做成一个**独立部署的通用钱包入账监控服务**，而不是耦合在某个业务系统内部。

核心目标：

- 系统维护一批需要监控的钱包地址；
- 周期性检查这些地址是否出现新的链上入账；
- 一旦识别到新入账，向业务方指定的服务端发起回调通知；
- 业务系统只负责“注册地址 + 接收回调”，不需要自己处理扫链逻辑。

## 2. 对话中明确的原始需求

根据图片对话，原始需求可以拆成 3 件事：

### 2.1 建表

需要使用 GORM 建立用于存储监控地址的数据表。

表结构至少应包含：

- 自增 ID
- 钱包地址字符串
- 时间字段
- 开关字段

### 2.2 监控数据库中的钱包地址

需要从数据库中读取所有待监控的钱包地址，并逐个检查是否有新的转账进入。

这部分的实际含义包括：

- 周期性扫描所有启用中的地址；
- 查询链上是否存在新的入账交易；
- 记录已经扫描到的区块高度或扫描进度；
- 避免对同一笔交易重复处理。

图片对话中提到“可能需要爬块”，说明真实生产方案需要对接链上数据源，按区块或按地址交易记录进行轮询。

### 2.3 入账后进行回调

当检测到地址有新的入账时，需要按约定格式发起 HTTP POST 回调到指定服务端。

回调至少应包含以下信息：

- 钱包地址
- 交易哈希
- 转出地址
- 转入地址
- 金额
- 区块高度

## 3. 当前代码已完成的部分

结合当前 `wallet_monitor` 代码，已经具备以下能力。

### 3.1 数据模型与建表

当前已通过 GORM 定义并自动迁移以下核心表：

- `WatchedAddress`：监控地址表
- `ProcessedTx`：已处理交易表，用于去重
- `MockIncomingTx`：本地联调用的 mock 入账表
- `ReceivedCallback`：本地调试回调记录表

这些表会在服务启动时通过 `AutoMigrate` 自动创建。

### 3.2 独立服务形态

当前实现已经是一个独立 HTTP 服务，而不是业务系统内嵌逻辑。

服务职责包括：

- 提供地址管理接口；
- 定时扫描监控地址；
- 发现入账后回调业务方；
- 提供本地联调用的 mock 接口与调试回调接口。

### 3.3 地址管理能力

当前已支持常用地址管理接口：

- `GET /addresses`：查询已注册的监控地址
- `POST /addresses`：新增监控地址
- `GET /addresses/{id}`：按 ID 查询
- `PATCH /addresses/{id}`：更新（`callback_url` / `enabled` / `min_confirmations` / `last_seen_height`）
- `DELETE /addresses/{id}`：删除监控地址
- `POST /addresses/{id}/enable`：启用监控
- `POST /addresses/{id}/disable`：禁用监控

并支持常用筛选参数（`GET /addresses?chain=...&network=...&asset_type=...&address=...&enabled=true|false`）。

当前 `WatchedAddress` 模型已包含以下主要字段：

- `ID`
- `Chain`
- `Network`
- `Address`
- `AssetType`
- `TokenContract`
- `CallbackURL`
- `Enabled`
- `MinConfirmations`
- `LastSeenHeight`
- `CreatedAt`
- `UpdatedAt`

这意味着“按链、网络、地址注册独立监控对象”的基础已经具备。

### 3.4 扫描框架

当前已实现完整的扫描框架：

- 定时任务循环扫描；
- 手动触发单次扫描；
- 逐个读取启用中的地址；
- 调用链适配器扫描是否有新入账；
- 扫描成功后更新 `last_seen_height`；
- 回调成功后记录交易处理结果。

新增的调试接口：

- `POST /scan/once`：手动执行一轮扫描，便于联调和测试

### 3.5 回调发送与去重

当前已经具备以下能力：

- 扫描阶段将回调写入 `CallbackTask`（持久化任务队列），避免“回调失败即丢单”；
- 回调发送由任务执行器处理，失败会按指数退避更新 `next_retry_at` 并记录 `last_error`；
- 超过最大重试次数后进入 `dead` 状态，可通过管理接口手动重试；
- 回调成功后写入 `ProcessedTx`，作为最终去重依据；
- 对 mock 交易进行“已消费”标记，避免重复触发。

同时支持可选回调签名（`-callback-secret`），便于业务方鉴权与验签。

说明：这一部分相比早期版本已经补齐，不能再归类为“完全未实现”。

### 3.6 本地闭环联调能力

当前服务已经支持**不接真实链 RPC 的本地端到端验证**。

新增调试接口：

- `GET/POST/DELETE /mock/transactions`：管理 mock 入账交易
- `GET/POST/DELETE /debug/callbacks`：查看和接收调试回调

通过把默认 `callback_url` 指向服务自身的 `/debug/callbacks`，可以在一台机器上完成以下闭环：

1. 注册监控地址；
2. 注入 mock 入账交易；
3. 手动触发扫描；
4. 验证是否成功写入回调记录；
5. 验证是否不会重复推送。

### 3.7 已落地的真实链能力

当前已经落地一版真实链适配器：

- 支持 `chain = tron`
- 支持 `network = mainnet | shasta | nile`
- 支持已确认的 `TRX` 入账扫描
- 支持已确认的 `TRC20` 入账扫描

实现方式：

- 使用 TronGrid/节点 HTTP API；
- 通过账户交易历史接口轮询指定地址的入账记录；
- 只识别成功且已确认的 `TransferContract` 交易；
- 对 `TRC20` 交易通过 `transactions/trc20` 接口查询，并补查交易详情拿到区块高度；
- 将扫到的新交易转换为统一的 `Tx` 结构后进入回调流程。

## 4. 当前仍未完成的核心工作

虽然当前服务已经具备完整框架和本地闭环能力，但离“真实生产可用的链上监控服务”还差几个关键点。

### 4.1 真实链适配器已部分完成，但范围还不够

这部分已经不是“完全未实现”，而是“部分可用”。

当前状态：

- `chain = mock`：可用于本地闭环测试；
- `chain = tron`：已支持真实链上的已确认 `TRX` / `TRC20` 入账扫描；
- 其他链（如 EVM、BTC 等）：仍未实现；
- 更复杂的 TRON 代币策略：仍未实现。

因此现在更准确的描述是：

- **本地服务链路可验证**
- **真实 TRON 主币 / TRC20 入账已可识别**
- **通用多链、多资产能力仍未完成**

后续需要补齐的内容包括：

- 扩展 EVM 等其他链的扫描适配器；
- 允许按币种 / 合约地址做更细粒度监控；
- 完善确认数和链重组处理策略。

### 4.2 入账判定规则需要业务明确

当前代码里还没有正式文档化以下业务规则：

- 是否只认主币转账；
- 是否要支持 TRC20 / ERC20 等代币转账；
- 是否只认成功上链交易；
- 是否需要等待若干确认数后再回调；
- 是否过滤零金额、失败交易、内部交易。

如果这些规则不明确，后续扩展 TRC20 和更多链时就很难做得稳定一致。

### 4.3 回调失败补偿机制不足

这部分已从“未实现”升级为“可用的生产最小闭环”：

- 已有失败记录表：`CallbackTask`
- 已有自动重试：`status=pending|retrying` + `next_retry_at` + 指数退避
- 已有最大重试次数与死信：`status=dead`
- 已有手动补发能力：支持按任务 ID 重试、或批量重试（见 `callback-tasks` 管理接口）

后续可继续增强的点（P1/P2）：

- 更细粒度的重试策略（按回调方错误码分类、熔断、并发控制）
- 更完善的死信处置流程（导出、告警、人工审核与补发工具）

### 4.4 地址管理能力还不完整

这部分已基本补齐（见 3.3），剩余更多是“运维体验增强”：

- 更丰富的查询与排序（按更新时间/高度/链等）
- 批量导入、批量启停与批量删除
- 变更审计（谁在什么时间改了 callback_url / enabled）

### 4.5 回调协议仍需正式文档化

当前代码里已经有回调结构，但还没有形成一份对外正式协议。

建议补充：

- 回调请求方法；
- 回调请求头；
- 回调 JSON 字段说明；
- 成功响应约定；
- 重试策略约定；
- 超时设置；
- 签名或鉴权机制。

### 4.6 可观测性仍需增强

当前已有基础日志和扫描结果统计，但还不够生产化。

建议继续补充：

- 每轮扫描耗时；
- 扫描地址数量；
- 命中的入账数量；
- 回调成功/失败数量；
- 当前扫描高度；
- Prometheus 指标；
- 异常告警。

### 4.7 配置与部署说明需要完善

当前已有命令行参数和测试文档，但部署说明还不完整。

建议补充：

- 服务启动方式；
- 参数说明；
- 数据库文件位置；
- RPC 地址配置方式；
- 默认回调地址配置方式；
- 扫描间隔建议；
- systemd / Docker 部署示例。

## 5. 建议的优先级

建议按以下顺序推进。

### P0

- 扩展真实链 `ScanAddress`
- 明确入账判定规则
- 保证不会漏单、不会重复回调

### P1

- 补齐回调失败重试机制
- 补齐地址管理接口
- 补充回调协议文档

### P2

- 增加指标、日志、告警
- 完善部署与运维说明

## 6. 结论

这份需求现在可以总结为：

1. 用数据库维护大量待监控钱包地址；
2. 周期性检查这些地址是否有新的入账；
3. 检测到入账后向指定服务发起 POST 回调；
4. 服务作为独立组件部署，与业务系统通过 HTTP 解耦。

结合当前代码，项目已经完成了：

- 表结构设计
- 独立服务骨架
- 基础地址管理接口
- 扫描框架
- 回调框架
- 交易去重
- 本地 mock 闭环测试能力
- TRON 主币入账扫描
- TRON TRC20 入账扫描

但它仍然**不是完整的生产版多链监控系统**，因为目前只完成了：

- `mock` 本地联调
- `tron` 的已确认 `TRX` / `TRC20` 入账扫描

尚未完成：

- ERC20 等其他链代币入账
- 多链扩展
- 回调失败重试机制
- 更完整的运维与可观测性能力

## 7. 方案判断：当前做法是否正确

### 7.1 结论

当前方案方向是对的。

更准确地说，当前实现属于：

- **按地址轮询 / 按地址查交易列表的第一版可用方案**；
- 适合先把“地址注册 → 扫描 → 去重 → 回调”这条主链路跑通；
- 对于当前已支持的 `tron + TRX` 场景，这种做法是合理的。

但需要明确：

- **“遍历数据库中的地址”是业务需求视角；**
- **“爬块”只是具体实现手段之一，不是唯一方案。**

因此，当前理解“读取所有地址，监控是否有变化，可能需要爬块”是正确的，但工程上应拆分为两类实现方式。

### 7.2 两类实现方式

#### 方案 A：按地址轮询（当前方案更接近这个）

流程：

1. 从数据库读取所有启用中的地址；
2. 针对每个地址调用链 RPC / 区块浏览器 API；
3. 查询该地址最近交易或余额变化；
4. 找到新的入账后做去重、更新扫描进度并回调。

优点：

- 实现简单；
- 易于快速上线；
- 适合单链、少量地址、MVP 阶段。

缺点：

- 地址量大时扫描成本高；
- 对 RPC 压力较大；
- 不利于后续扩展多链、多资产；
- 在 reorg、确认数推进等场景下扩展成本更高。

#### 方案 B：按区块高度爬块（更接近生产型索引方案）

流程：

1. 服务维护全局扫描高度；
2. 周期性拉取新区块或新区块中的转账 / 日志；
3. 判断命中的收款地址是否在监控地址集合中；
4. 命中后更新进度、幂等写入并回调。

优点：

- 更适合大量地址；
- 更容易统一处理确认数、reorg 回滚、事件重放；
- 更适合演进成通用链上索引服务。

缺点：

- 实现复杂度更高；
- 不同链的适配差异较大；
- 对区块/日志解析能力要求更高。

### 7.3 当前代码属于哪一种

结合当前实现，项目目前更接近：

- **方案 A：按地址轮询交易列表 / 地址扫描方案**。

依据：

- 当前 `tron` 适配器是通过 TronGrid HTTP API 查询指定地址的交易历史；
- 然后筛选成功且已确认的 `TRX` 入账；
- 再转换成统一 `Tx` 结构进入回调流程。

也就是说，现在并不是“全链统一爬块索引器”，而是“地址级扫描器 + 链适配器”的架构。

### 7.4 现阶段是否需要改成爬块

不一定立刻需要。

建议按阶段推进：

#### 阶段 1：先保留当前地址扫描方案

适用条件：

- 监控地址数量不大；
- 先验证业务闭环；
- 当前重点是把真实链、去重、回调、确认数跑通。

这一阶段，当前方案是正确的。

#### 阶段 2：地址量增长后，再演进为爬块 / 索引方案

适用条件：

- 地址规模明显上升；
- 单地址轮询带来较大 RPC 压力；
- 需要支持多链、多资产（TRC20/ERC20）；
- 需要更稳定地处理 reorg、确认数和批量对账。

这一阶段，建议逐步演进为：

- 全局扫描高度；
- 区块 / 日志驱动；
- 命中监控地址后回调；
- 配套确认数推进、回滚补偿、失败重试与指标监控。

### 7.5 最终判断

因此，这个需求目前可以统一理解为：

1. 数据库维护一批待监控地址；
2. 服务周期性检测这些地址是否产生新的链上入账；
3. 检测方式可以是：
   - 按地址查询交易 / 余额；
   - 或按区块高度爬块匹配地址；
4. 一旦识别到新入账，进行幂等处理并向业务方发起回调。

**结论：当前方案是对的，但它是第一版可用方案，不是最终的生产级多链索引方案。**

后续是否升级为“爬块方案”，主要取决于：

- 地址规模；
- 链类型数量；
- 资产种类（主币 / TRC20 / ERC20）；
- 对确认数、reorg、性能与可观测性的要求。
